# SIXTH_STEP — Embedding Caching & New Clusterers — 2026-04-04

## What was built

Two pieces of work: an SQLite-backed embedding cache with lazy model loading, and two new HDBSCAN clusterer variants that skip or reorder the UMAP step.

---

## 1. Embedding Cache

### Problem

Every pipeline run loaded the `all-MiniLM-L6-v2` model (~80MB) and re-computed embeddings for all source texts and theme texts from scratch, even when running the same input repeatedly. This made iteration slow.

### Solution

Cache embeddings by exact text on disk. If all texts are cached, the embedding model is never loaded at all.

### New files

- **`src/embedding_cache.py`** — `EmbeddingCache` class backed by SQLite (Python built-in `sqlite3`, no new dependencies).
  - Table schema: `cache_key` (SHA-256 of `model_name + "\0" + text`), `model_name`, `text`, `embedding` (BLOB as float32 bytes), `created_at`
  - Batch operations: `get_many(texts)` and `put_many(entries)` for efficient lookups/inserts
  - WAL journal mode for performance
  - Default location: `.cache/embeddings.db`
- **`tests/test_embedding_cache.py`** — 9 unit tests covering roundtrips, partial hits, cross-model isolation, persistence across instances, clear, and edge cases
- **`tests/test_embedder_cache.py`** — 7 integration tests verifying lazy model loading, full cache hits skipping model load, partial cache encoding only uncached texts, `--no-cache` bypass, persistence of new embeddings, empty input, and duplicate texts

### Modified files

- **`src/embedder.py`** — `SentenceTransformerEmbedder` restructured:
  - Constructor no longer loads the model. Accepts `cache_dir` and `no_cache` params.
  - `_get_model()` lazy-loads `SentenceTransformer` on first call (import is also deferred)
  - `embed()` checks cache first, splits texts into cached/uncached, only loads the model if there are uncached texts, then persists new embeddings to cache
- **`run.py`** — Added `--no-cache` (disable cache) and `--cache-dir` (override default `.cache`) CLI arguments, passed to embedder constructor
- **`.gitignore`** — Added `.cache/` to prevent the SQLite database from being committed

### Design decisions

- **SQLite over JSON/shelve**: Atomic writes (no corruption from interrupted runs), efficient key lookups at scale, single file, built-in to Python
- **Cache key is SHA-256 hash**: Avoids SQLite performance degradation with very long text strings as primary keys. Full text stored in the row for debuggability.
- **Model name in cache key**: Same text with different models produces separate cache entries, preventing stale embeddings if the model changes
- **Lazy import of `sentence_transformers`**: The `from sentence_transformers import SentenceTransformer` is inside `_get_model()`, so even the import is skipped when all texts are cached

### No changes needed to

- `src/pipeline.py` — The `Embedder` protocol (`embed(chunks) -> chunks`) is unchanged
- `src/steerers/` and `src/clusterers/` — They call `embedder.embed()` which handles caching transparently
- Existing tests — `MockEmbedder` doesn't touch caching, all 39 prior tests pass unchanged

## 2. New Clusterers

### Motivation

The default `umap_hdbscan` clusterer reduces embeddings from 384→10 dimensions with UMAP *before* HDBSCAN clusters them. This is standard practice (HDBSCAN can struggle with high-dimensional density estimation), but it means the UMAP step can partially wash out the directional amplifications applied by the projection steerer. Two new clusterers offer alternatives.

### New files

- **`src/clusterers/hdbscan_direct.py`** — `HDBSCANDirectClusterer`: runs HDBSCAN directly on full 384-dim embeddings with no UMAP at all. The steering signal is fully preserved for clustering. UMAP is only used in `reduce_for_viz()` for the HTML visualization.
- **`src/clusterers/hdbscan_umap.py`** — `HDBSCANUMAPClusterer`: runs HDBSCAN first on full 384-dim embeddings, then applies UMAP reduction afterward. Clustering sees full signal; UMAP only reduces embeddings post-hoc (replacing chunk embeddings with the lower-dimensional versions for downstream use).
- **`tests/test_clusterers.py`** — 12 tests across both clusterers and module loading: cluster output structure, too-few-chunks fallback, chunk object identity, viz reduction shape, embedding preservation (direct) / embedding reduction (umap), and dynamic module loading

### Usage

```bash
# HDBSCAN on full 384-dim embeddings (no UMAP)
python run.py data/sample_input.jsonl --clusterer hdbscan_direct --cluster-only

# HDBSCAN first, then UMAP reduction
python run.py data/sample_input.jsonl --clusterer hdbscan_umap --cluster-only
```

Both support `--min-cluster-size` (default 3), same as the original `umap_hdbscan`.

### Tradeoffs

| Clusterer | HDBSCAN input | Steering preserved | Dimensionality curse risk |
|---|---|---|---|
| `umap_hdbscan` (default) | 10-dim UMAP | Partially (UMAP is lossy) | Low |
| `hdbscan_direct` | 384-dim raw | Fully | Higher |
| `hdbscan_umap` | 384-dim raw | Fully (for clustering) | Higher |

The right choice depends on the data. A/B comparison across clusterers on steered input is the way to determine which produces more meaningful clusters for a given dataset.
